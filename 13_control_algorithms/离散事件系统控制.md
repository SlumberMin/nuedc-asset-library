# 离散事件系统控制指南

> 适用场景：电赛中涉及多任务调度、资源分配、状态机设计、排队/调度优化的系统

## 一、离散事件系统（DES）概述

### 1.1 连续 vs 离散事件系统

| 特征 | 连续系统 | 离散事件系统 |
|------|---------|-------------|
| 状态 | 连续值 | 有限离散状态 |
| 时间驱动 | 微分方程 | 事件触发 |
| 变化方式 | 平滑变化 | 瞬时跳变 |
| 建模工具 | 传递函数/状态空间 | 自动机/Petri网 |
| 典型应用 | 电机/温度控制 | 交通灯/多任务调度 |

### 1.2 电赛中的DES场景

```
- 多模块协调工作（传感器采集 + 电机控制 + 显示）
- 竞赛任务分阶段执行（寻迹 → 避障 → 投放 → 返回）
- 资源仲裁（多路ADC分时复用、通信总线仲裁）
- 故障检测与恢复（超时重试、降级运行）
```

## 二、有限状态机（FSM）

### 2.1 状态机基本概念

```
FSM = (Q, Σ, δ, q₀, F)
  Q：有限状态集合
  Σ：输入事件集合
  δ：状态转移函数 Q × Σ → Q
  q₀：初始状态
  F：终止状态集
```

### 2.2 Mealy vs Moore

```
Moore机：输出只依赖当前状态
  输出 = f(状态)
  → 输出更稳定，但需要更多状态

Mealy机：输出依赖状态和输入
  输出 = f(状态, 输入)
  → 响应更快，但可能有毛刺
```

### 2.3 电赛状态机设计模板（C语言）

```c
// 事件定义
typedef enum {
    EVT_TIMEOUT, EVT_SENSOR_TRIG, EVT_BUTTON,
    EVT_OBSTACLE, EVT_ARRIVED, EVT_ERROR
} Event_t;

// 状态定义
typedef enum {
    ST_IDLE, ST_SEARCHING, ST_AVOIDING,
    ST_CARRYING, ST_RETURNING, ST_ERROR
} State_t;

// 状态处理函数类型
typedef State_t (*StateHandler_t)(Event_t evt);

// 状态转移表
State_t State_Idle(Event_t evt) {
    switch(evt) {
        case EVT_BUTTON:   return ST_SEARCHING;
        case EVT_ERROR:    return ST_ERROR;
        default:           return ST_IDLE;
    }
}

State_t State_Searching(Event_t evt) {
    switch(evt) {
        case EVT_OBSTACLE: return ST_AVOIDING;
        case EVT_ARRIVED:  return ST_CARRYING;
        case EVT_TIMEOUT:  return ST_RETURNING;
        case EVT_ERROR:    return ST_ERROR;
        default:           return ST_SEARCHING;
    }
}

// ... 其他状态处理函数

// 状态机主循环
StateHandler_t state_table[] = {
    State_Idle, State_Searching, State_Avoiding,
    State_Carrying, State_Returning, State_Error
};

State_t current_state = ST_IDLE;

void FSM_Dispatch(Event_t evt) {
    State_t next = state_table[current_state](evt);
    if(next != current_state) {
        // 执行退出动作
        OnExit[current_state]();
        // 执行进入动作
        OnEnter[next]();
        current_state = next;
    }
}
```

### 2.4 层次状态机（HSM）

```
适合复杂系统，状态可嵌套：

          ┌──────────────────┐
          │     运行(RUNNING)  │ ← 超状态
          │ ┌──────┐ ┌──────┐│
          │ │ 寻迹  │→│ 避障  ││ ← 子状态
          │ └──┬───┘ └──────┘│
          │    ↓              │
          │ ┌──────┐          │
          │ │ 投放  │          │
          │ └──────┘          │
          └──────────────────┘
               ↑ ↓
          ┌──────────┐
          │  停止(STOPPED) │
          └──────────┘

退出RUNNING时自动退出当前子状态
```

## 三、Petri网

### 3.1 Petri网基本概念

```
Petri网 = (P, T, F, M₀)
  P：库所（Place）集合 → 用圆圈○表示，含令牌(token)
  T：变迁（Transition）集合 → 用方块■表示
  F：流关系 → 有向弧
  M₀：初始标识（各库所的初始令牌数）

变迁触发规则：
  1. 前提条件：输入库所的令牌数 ≥ 弧的权重
  2. 效果：消耗输入库所令牌，产生输出库所令牌
```

### 3.2 Petri网图形表示

```
简单例子：生产-消费模型

  P1(生产就绪) ──→ T1(生产) ──→ P2(缓冲区) ──→ T2(消费) ──→ P3(完成)
                   ↑                                                │
                   └────────────────────────────────────────────────┘

  初始标识：P1=1, P2=0, P3=0
```

### 3.3 Petri网性质分析

| 性质 | 含义 | 电赛意义 |
|------|------|---------|
| 可达性 | 能否从M₀到达某状态 | 系统能否完成任务 |
| 有界性 | 令牌数有上界 | 缓冲区不会溢出 |
| 活性 | 无死锁 | 系统不会卡死 |
| 可逆性 | 能回到初始状态 | 系统可复位 |

### 3.4 Petri网建模实例：电赛多任务

```
建模：传感器采集 + 电机控制 + 显示更新 的并发协调

库所：
  P1=传感器就绪  P2=数据待处理  P3=电机就绪
  P4=电机运行中  P5=显示就绪    P6=显示缓冲
  P7=采集中      P8=控制中

变迁：
  T1=启动采集  T2=采集完成  T3=启动控制
  T4=控制完成  T5=更新显示

                T1         T2
  P1(传感器就绪)→P7(采集中)→P2(数据待处理)
                                  ↓ T3
  P3(电机就绪)──────────→P8(控制中)→T4→P3(电机就绪)
                                               ↓
  P5(显示就绪)←──T5←──P6(显示缓冲)←───┘
```

### 3.5 互斥与死锁

```
互斥建模（共享资源）：

  P_res(资源可用，令牌=1)
       ↓          ↓
  T1(请求1)    T2(请求2)
       ↓          ↓
  P1_use      P2_use
       ↓          ↓
  T1_rel      T2_rel
       ↓          ↓
       └→ P_res ←─┘

→ 自动保证互斥（只有1个令牌）
```

```
死锁避免：
  危险：循环等待
  预防策略：
    1. 银行家算法：预判分配后是否仍安全
    2. 资源有序分配：按固定顺序请求资源
    3. 超时机制：等待超时后释放已占资源
```

## 四、排队论基础

### 4.1 排队系统模型

```
Kendall记号：A/B/c/K/N/D
  A：到达间隔分布（M=泊松, D=确定, G=一般）
  B：服务时间分布
  c：服务台数量
  K：系统容量（默认∞）
  N：顾客源大小（默认∞）
  D：排队规则（FIFO/LIFO/优先级）

常见模型：
  M/M/1：单服务台，泊松到达，指数服务
  M/M/c：多服务台
  M/D/1：确定性服务时间
```

### 4.2 M/M/1核心公式

```
设：到达率λ，服务率μ，利用率ρ=λ/μ

平均队长：L = ρ/(1-ρ)
平均等待：W = 1/(μ-λ)
平均逗留：L_s = λ/(μ-λ)
空闲概率：P₀ = 1-ρ
```

### 4.3 电赛排队论应用

```
场景：多路传感器数据经共享ADC采集

传感器到达率：λ = 100次/秒（总）
ADC处理率：μ = 200次/秒
利用率：ρ = 100/200 = 0.5

平均等待时间：W = 1/(200-100) = 10ms
平均队长：L = 0.5/(1-0.5) = 1个

→ 可接受（等待时间<采样周期）
```

```c
// 排队论辅助：计算系统是否过载
float calc_utilization(float arrival_rate, float service_rate) {
    return arrival_rate / service_rate;
}

// 如果ρ > 0.8，系统接近饱和，需要：
// 1. 提高服务率（加快ADC/通信速率）
// 2. 降低到达率（降低采样频率）
// 3. 增加服务台（多ADC并行）
```

## 五、离散事件仿真

### 5.1 事件驱动仿真框架

```c
// 事件结构
typedef struct {
    float time;         // 事件时间
    int type;           // 事件类型
    int source;         // 事件来源
    void *data;         // 事件数据
} Event_t;

// 事件队列（优先队列/最小堆）
Event_t event_queue[MAX_EVENTS];
int queue_size = 0;

// 仿真主循环
void DES_Simulate(float end_time) {
    while(queue_size > 0 && current_time < end_time) {
        Event_t evt = pop_min(event_queue);  // 取最早事件
        current_time = evt.time;
        process_event(&evt);                  // 处理事件（可能产生新事件）
    }
}

// 产生新事件
void schedule_event(float time, int type, int source, void *data) {
    Event_t new_evt = {time, type, source, data};
    push(event_queue, new_evt);
}
```

### 5.2 状态图到代码的映射

```
状态图元素 → C语言实现：

转移    → switch-case分支

动作    → 函数调用
```

## 六、电赛实战：竞赛任务调度系统

### 6.1 任务调度状态机

```c
// 竞赛任务：小车巡线→抓取→搬运→放置→返回

typedef enum {
    TASK_IDLE, TASK_PATROL, TASK_GRAB,
    TASK_TRANSPORT, TASK_PLACE, TASK_RETURN, TASK_DONE
} TaskState_t;

typedef enum {
    TRIG_START, TRIG_LINE_END, TRIG_OBJECT_FOUND,
    TRIG_GRABBED, TRIG_ARRIVED, TRIG_PLACED,
    TRIG_HOME, TRIG_TIMEOUT
} Trigger_t;

TaskState_t task_fsm[][8] = {
    // 状态转移表 [当前状态][触发信号] → 下一状态
    //  TRIG_START    LINE_END     OBJ_FOUND    GRABBED      ARRIVED      PLACED       HOME         TIMEOUT
    {  TASK_PATROL,  TASK_IDLE,   TASK_IDLE,   TASK_IDLE,   TASK_IDLE,   TASK_IDLE,   TASK_IDLE,   TASK_IDLE  }, // IDLE
    {  TASK_PATROL,  TASK_PATROL, TASK_GRAB,   TASK_PATROL, TASK_PATROL, TASK_PATROL, TASK_PATROL, TASK_RETURN}, // PATROL
    {  TASK_GRAB,    TASK_GRAB,   TASK_GRAB,   TASK_TRANSPORT,TASK_GRAB, TASK_GRAB,   TASK_GRAB,   TASK_RETURN}, // GRAB
    {  TASK_TRANSPORT,TASK_TRANSPORT,TASK_TRANSPORT,TASK_TRANSPORT,TASK_PLACE,TASK_TRANSPORT,TASK_TRANSPORT,TASK_RETURN}, // TRANSPORT
    {  TASK_PLACE,   TASK_PLACE,  TASK_PLACE,  TASK_PLACE,  TASK_PLACE,  TASK_RETURN, TASK_PLACE,  TASK_RETURN}, // PLACE
    {  TASK_RETURN,  TASK_RETURN, TASK_RETURN, TASK_RETURN, TASK_RETURN, TASK_RETURN, TASK_DONE,   TASK_DONE  }, // RETURN
    {  TASK_DONE,    TASK_DONE,   TASK_DONE,   TASK_DONE,   TASK_DONE,   TASK_DONE,   TASK_DONE,   TASK_DONE  }, // DONE
};

TaskState_t current_task = TASK_IDLE;
```

### 6.2 超时看门狗集成

```c
// 每个任务阶段设置超时
typedef struct {
    TaskState_t state;
    float timeout_ms;
} TaskTimeout_t;

TaskTimeout_t timeouts[] = {
    {TASK_PATROL,    30000},  // 巡线30秒
    {TASK_GRAB,      5000},   // 抓取5秒
    {TASK_TRANSPORT, 20000},  // 搬运20秒
    {TASK_PLACE,     5000},   // 放置5秒
    {TASK_RETURN,    30000},  // 返回30秒
};

void check_timeout(void) {
    if(task_timer > get_timeout(current_task)) {
        FSM_Dispatch(TRIG_TIMEOUT);  // 触发超时事件
    }
}
```

## 七、选择指南

```
问题类型 → 推荐工具：

简单多状态切换 → FSM（有限状态机）
复杂嵌套状态  → HSM（层次状态机）
并发/同步/资源竞争 → Petri网
系统容量/性能评估 → 排队论
需要验证性质（死锁等） → Petri网 + 模型检查

电赛最常用：FSM > HSM > 排队论 > Petri网
```
