## 1. 系统卡顿

### 1.1 MCU系统卡顿

| 现象 | 可能原因 | 解决方案 |
|------|----------|----------|
| 主循环执行极慢 | 中断过于频繁 | 减少不必要的中断/降低中断频率 |
| 定时器回调执行过长 | 回调中做了重计算 | 回调只设标志位，主循环处理 |
| while死循环卡死 | 条件永远不满足 | 加超时退出机制 |
| DMA传输阻塞CPU | DMA配置不当 | 使用DMA中断回调模式 |

### 1.2 上位机(树莓派/Jetson)卡顿

| 现象 | 可能原因 | 解决方案 |
|------|----------|----------|
| 系统响应慢 | CPU占用100% | 用top/htop查看占用最高的进程 |
| 图像处理卡顿 | 内存不足/swap | 关闭不需要的服务、增大swap |
| SSH连接卡死 | 温度过高降频 | 加散热风扇、监控温度 |
| 进程无响应 | 死锁/阻塞I/O | 检查线程同步、加超时机制 |

### 1.3 性能优化策略

```
MCU层面:
  - 检查所有中断优先级分配是否合理
  - 高频中断中只做数据搬运，不做计算
  - 使用DMA替代CPU搬运
  - 关闭不使用的外设时钟
  - 使用编译器优化选项(-O2/-Os)

Linux上位机层面:
  - 使用nice/renice调整进程优先级
  - 关闭不需要的systemd服务
  - 使用taskset绑定CPU核心
  - 减少不必要的日志写入(磁盘I/O)
  - 使用内存映射文件替代频繁文件读写
```

## 2. 进程/任务崩溃

### 2.1 MCU崩溃(HardFault)

| 现象 | 可能原因 | 解决方案 |
|------|----------|----------|
| 进入HardFault_Handler | 空指针/数组越界 | 查看LR和PC寄存器定位崩溃位置 |
| 栈溢出导致崩溃 | 局部变量过大/递归过深 | 增大栈空间、减少局部变量 |
| 未处理中断 | 中断使能但无回调 | 检查所有中断向量表 |
| 看门狗复位 | 主循环卡死未喂狗 | 排查主循环阻塞点 |

### 2.2 HardFault定位方法

```c
// 方法1: 在HardFault_Handler中读寄存器
void HardFault_Handler(void) {
    __asm volatile (
        "TST LR, #4       \n"
        "ITE EQ           \n"
        "MRSEQ R0, MSP    \n"
        "MRSNE R0, PSP    \n"
        "B hard_fault_handler_c \n"
    );
}

void hard_fault_handler_c(uint32_t *stack) {
    uint32_t pc = stack[6];  // 崩溃时的PC值
    uint32_t lr = stack[5];  // 崩溃时的LR值
    // pc即为崩溃地址，查map文件定位函数
    while(1);
}

// 方法2: 使用J-Link的HardFault分析功能
// 方法3: 开启MPU保护，在越界时触发MemManage
```

### 2.3 Linux进程崩溃

| 现象 | 可能原因 | 解决方案 |
|------|----------|----------|
| Segmentation fault | 空指针/越界访问 | 用gdb调试、加-G编译选项 |
| Bus error | 内存对齐问题 | 检查结构体对齐 |
| Killed (OOM) | 内存耗尽 | 检查内存泄漏、减小缓冲区 |
| 核心已转储 | 未捕获异常 | 设置信号处理器、检查日志 |

## 3. 内存问题

### 3.1 内存泄漏

| 场景 | 检测方法 | 解决方案 |
|------|----------|----------|
| C/C++ malloc未free | Valgrind/AddressSanitizer | 统一内存管理/智能指针 |
| Python对象未释放 | tracemalloc/gc | 及时del引用/使用with语句 |
| OpenCV Mat未释放 | 监控进程内存 | 及时release/赋值空 |

### 3.2 栈溢出

| 现象 | 原因 | 解决方案 |
|------|------|----------|
| 变量莫名被改写 | 栈空间不足 | 增大栈空间、减少局部大数组 |
| 函数返回异常 | 栈被覆盖 | 使用静态数组替代栈上大数组 |
| 系统随机崩溃 | 递归过深 | 改用迭代实现 |

### 3.3 内存优化技巧

```
MCU:
  - 大数组声明为static或全局(不占栈)
  - 使用__attribute__((aligned(4)))确保对齐
  - 使用const放在Flash而非RAM
  - 编译后查看.map文件分析内存占用

Linux:
  - 使用mmap替代malloc处理大文件
  - 及时释放OpenCV Mat/Python大对象
  - 使用内存池减少碎片化
  - 监控: watch -n 1 'free -m'
```

## 4. 多任务/多线程问题

### 4.1 RTOS(FreeRTOS)问题

| 现象 | 可能原因 | 解决方案 |
|------|----------|----------|
| 任务不调度 | 优先级配置错误 | 检查各任务优先级 |
| 优先级反转 | 低优先级任务持锁 | 使用互斥量(带优先级继承) |
| 死锁 | 循环等待 | 统一加锁顺序/使用超时 |
| 任务栈溢出 | 栈分配不足 | 开启栈溢出检测，增大栈 |

### 4.2 Linux多线程问题

| 现象 | 可能原因 | 解决方案 |
|------|----------|----------|
| 数据竞争 | 共享变量未保护 | 使用mutex/atomic |
| 死锁 | 多锁交叉等待 | 统一锁序/使用try_lock+超时 |
| 线程不退出 | join阻塞 | 设置退出标志+detach |

### 4.3 嵌入式临界区保护

```c
// 方式1: 关中断
taskENTER_CRITICAL();
// 临界区代码
taskEXIT_CRITICAL();

// 方式2: 使用互斥量
xSemaphoreTake(mutex, portMAX_DELAY);
// 临界区代码
xSemaphoreGive(mutex);

// 方式3: 裸机中使用volatile+关中断
__disable_irq();
shared_data = new_value;
__enable_irq();
```

## 5. 文件系统/I/O问题

### 5.1 SD卡/存储问题

| 现象 | 可能原因 | 解决方案 |
|------|----------|----------|
| SD卡无法挂载 | 文件系统损坏 | 格式化为FAT32/exFAT |
| 写入速度慢 | SD卡速度等级低 | 使用Class10/UHS-I以上 |
| 数据丢失 | 未正常卸载就断电 | 使用UPS/先同步再断电 |
| SPI模式SD卡不稳定 | SPI时钟过快 | 降低SPI频率、加100kΩ上拉 |

### 5.2 串口日志问题

| 现象 | 原因 | 解决方案 |
|------|------|----------|
| printf阻塞导致卡顿 | 串口缓冲区满 | 使用DMA发送或增大缓冲区 |
| 日志丢失 | 输出速度>传输速度 | 降低日志频率/增大缓冲 |
| 格式化输出异常 | 格式符与类型不匹配 | 检查%d/%f/%s与变量类型 |

## 7. 实战案例库

### 案例1：串口通信丢包的解决方案

*问题现象*：2025年E题中，Orange Pi 5与STM32间UART通信出现丢包，导致云台控制不连续。

*问题分析*：
```
现象记录:
  - 波特率115200, 数据包20字节
  - 丢包率: 5-10%
  - 高丢包时刻: CPU占用>70%
  - 接收端缓冲区溢出日志
```

*根本原因*：
1. STM32接收中断优先级太低，被其他中断抢占
2. Orange Pi发送速率过快，STM32来不及处理
3. 没有流控机制

*解决方案*：
```c
// STM32端：提高UART中断优先级
void UART_Init(void) {
    // UART2中断优先级设为最高
    HAL_NVIC_SetPriority(UART2_IRQn, 0, 0);  // 优先级0，最高
    HAL_NVIC_EnableIRQ(UART2_IRQn);

    // 增大接收缓冲区
    #define RX_BUF_SIZE 256
    uint8_t rx_buffer[RX_BUF_SIZE];
}

// 协议设计：增加帧头、长度、校验
#pragma pack(push, 1)
typedef struct {
    uint8_t header;      // 0xAA
    uint8_t length;      // 数据长度
    uint8_t data[20];    // 数据域
    uint8_t checksum;    // 校验和
    uint8_t tail;        // 0x55
} UART_Packet_t;
#pragma pack(pop)

// 校验和计算
uint8_t Calculate_Checksum(uint8_t *data, uint8_t len) {
    uint8_t sum = 0;
    for (int i = 0; i < len; i++) sum ^= data[i];
    return sum;
}
```

```python
# Orange Pi端：加入流控和重发机制
class ReliableUART:
    def __init__(self, port, baud=115200):
        self.ser = serial.Serial(port, baud, timeout=0.1)
        self.seq_num = 0
        self.tx_queue = queue.Queue(maxsize=100)

    def send_packet(self, data):
        """可靠发送"""
        packet = self.build_packet(data)
        retries = 3
        while retries > 0:
            self.ser.write(packet)
            # 等待ACK（简化版）
            time.sleep(0.01)
            retries -= 1
            break  # 简化：不做ACK等待

    def build_packet(self, data):
        packet = bytearray()
        packet.append(0xAA)  # 帧头
        packet.append(len(data))
        packet.extend(data)
        checksum = self.calc_checksum(data)
        packet.append(checksum)
        packet.append(0x55)  # 帧尾
        return bytes(packet)
```

*效果*：丢包率降至0.1%以下，云台控制平滑。

### 案例2：中断优先级配置不当导致的系统崩溃

*问题现象*：STM32在高负载时进入HardFault，系统崩溃。

*问题分析*：
```
崩溃现场:
  - PC寄存器: 0x08001234 (在SysTick_Handler中)
  - LR寄存器: 0xFFFFFFFD (PSP返回)
  - 崩溃时刻: 正在处理TIM3中断
```

*根本原因*：
```
中断优先级冲突:
  - SysTick: Priority 0 (最高)
  - TIM3 (电机控制): Priority 1
  - UART2 (通信): Priority 2

问题: TIM3中断中调用的函数触发了UART传输
     而UART中断优先级更低，导致死锁
```

*解决方案*：
```c
// 正确的中断优先级规划
void Interrupt_Priority_Config(void) {
    // 优先级0 (最高): 紧急保护
    HAL_NVIC_SetPriority(BUS_FAULT_IRQn, 0, 0);

    // 优先级1: 实时控制
    HAL_NVIC_SetPriority(TIM3_IRQn, 1, 0);  // 电机PWM
    HAL_NVIC_SetPriority(ADC_IRQn, 1, 1);   // 传感器采样

    // 优先级2: 通信
    HAL_NVIC_SetPriority(UART2_IRQn, 2, 0);
    HAL_NVIC_SetPriority(I2C1_IRQn, 2, 1);

    // 优先级3 (最低): 非实时
    HAL_NVIC_SetPriority(SysTick_IRQn, 3, 0);

    // 关键: 同优先级内不要有依赖关系
}

// 避免在低优先级中断中调用高优先级中断的阻塞函数
void TIM3_IRQHandler(void) {
    // 正确: 只设置标志位
    motor_update_flag = 1;
}

// 主循环中处理
int main(void) {
    while (1) {
        if (motor_update_flag) {
            Motor_Update();  // 主循环中处理
            motor_update_flag = 0;
        }
    }
}
```

*效果*：HardFault完全消除，系统稳定运行。

### 案例3：电源纹波对ADC采样的影响

*问题现象*：ADC读数波动大（±20个LSB），影响磁悬浮高度测量精度。

*问题分析*：
```
测试数据:
  - 期望值: 2048 (中间值)
  - 实际波动: 2028 ~ 2068
  - 纹波频率: 20kHz (与PWM频率相同)
  - 波形: 锯齿波叠加高频噪声
```

*根本原因*：
1. 电磁铁PWM驱动产生20kHz开关噪声
2. ADC采样时钟与PWM不同步
3. 电源线和信号线平行走线，耦合干扰

*解决方案*：
```c
// 硬件改进
// 1. 分离模拟地和数字地
// 2. ADC供电加LC滤波
// 3. 模拟信号线加屏蔽

// 软件改进：多采样+数字滤波
#define ADC_OVERSAMPLE 16

uint16_t ADC_ReadFiltered(uint8_t channel) {
    uint32_t sum = 0;
    for (int i = 0; i < ADC_OVERSAMPLE; i++) {
        sum += ADC_Read(channel);
        delay_us(10);  // 等待PWM周期结束
    }
    return sum / ADC_OVERSAMPLE;  // 均值滤波
}

// 更优：中值滤波（去除脉冲干扰）
uint16_t ADC_ReadMedianFiltered(uint8_t channel) {
    uint16_t samples[ADC_OVERSAMPLE];
    for (int i = 0; i < ADC_OVERSAMPLE; i++) {
        samples[i] = ADC_Read(channel);
    }
    // 排序取中值
    sort(samples, ADC_OVERSAMPLE);
    return samples[ADC_OVERSAMPLE / 2];
}

// 软件低通滤波
typedef struct {
    float alpha;  // 滤波系数 (0.0-1.0)
    float value;
} LowPassFilter;

float LowPass_Update(LowPassFilter *f, float new_value) {
    f->value = f->alpha * new_value + (1 - f->alpha) * f->value;
    return f->value;
}

// 使用: alpha = 0.1 (强滤波), 0.3 (中等), 0.7 (弱滤波)
```

*效果*：ADC波动从±20降至±3个LSB，高度测量精度提升6倍。
