## 第1天：选题与采购阶段

### 应急情况1: 题目涉及未准备过的模块

**应对方案:**
```
1. 不要慌，先看清楚基本要求和发挥部分
2. 基本要求优先：用最熟悉的技术栈完成基本功能
3. 发挥部分：评估难度，挑最有把握的1-2项做
4. 立即搜索类似题目往届方案(电赛公众号/论坛)
5. 器件采购：列清单→本地电子市场/快递→优先买核心模块
```

### 应急情况2: 关键器件买不到

**应对方案:**
```
备选方案矩阵:
  缺摄像头 → 用红外对管阵列替代(降级方案)
  缺电机驱动 → 用MOS管搭H桥
  缺传感器模块 → 用分立元件+运放搭电路
  缺开发板 → 用最小系统板飞线
  
应急采购渠道:
  - 本地电子市场(周末可能关门，提前确认)
  - 淘宝同城闪送(当天到)
  - 队友/同学之间借调
```

## 第2天：搭建与调试阶段

### 应急情况3: 焊接失误导致PCB/模块烧毁

**应对方案:**
```
1. 立即断电
2. 检查烧毁范围(只烧芯片还是整板)
3. 单芯片烧毁 → 飞线替换模块
4. 整板烧毁 → 切换到备用方案(面包板/洞洞板搭电路)
5. 【预防】准备2套核心模块，至少1套备份
```

### 应急情况4: 代码编译通过但系统不工作

**应对方案:**
```
排查顺序(5分钟法则):
  1. 电源 → 用万用表测各点电压 (1分钟)
  2. 时钟 → 示波器看晶振是否起振 (1分钟)
  3. 下载 → 重新下载程序确认烧录成功 (1分钟)
  4. GPIO → 点灯测试确认程序在跑 (1分钟)
  5. 通信 → 用串口打印确认关键变量值 (1分钟)

如果以上都正常，逐步加模块调试:
  先跑空循环 → 加一个模块 → 测试通过 → 加下一个模块
```

### 应急情况5: 电机/舵机不响应

**应对方案:**
```
快速排查:
  1. 独立供电测试(排除软件问题)
  2. PWM信号用示波器确认(频率+占空比)
  3. 驱动板输出用万用表确认
  4. 电机本体用电池直接供电测试

应急替代:
  舵机抖动 → 加100μF电容+独立5V供电
  直流电机不转 → 检查驱动MOS是否烧毁，用继电器替代
  步进电机丢步 → 降低速度、增大电流
```

## 第3天：联调与优化阶段

### 应急情况6: 控制算法效果差

**应对方案:**
```
降级方案(按优先级):
  1. PID振荡 → 大幅降低Kp，接受慢响应
  2. 开环替代 → 用查表法+开环控制替代复杂闭环
  3. 简化模型 → 线性近似替代非线性模型
  4. 分段控制 → 不同区间用不同参数

实用技巧:
  - 先用示波器看控制量波形
  - 记录输入输出数据，离线分析
  - 不要在比赛中尝试新算法，用验证过的代码
```

### 应急情况7: 视觉识别不稳定

**应对方案:**
```
快速补救:
  1. 降低识别要求(只识别最显著的特征)
  2. 加大颜色阈值容差
  3. 增加帧间滤波(连续N帧检测到才确认)
  4. 改用简单模板匹配替代复杂算法
  5. 加物理辅助(遮光罩、补光灯、纯色背景布)

终极降级:
  - 纯色目标用二值化+面积判断(最稳定)
  - 矩形目标用边缘检测+轮廓拟合
  - 放弃精确识别，用位置记忆+预测替代
```

### 应急情况8: 通信模块频繁断连

**应对方案:**
```
蓝牙断连:
  1. 缩短通信距离(<2米)
  2. 降低数据发送频率
  3. 改用有线串口(USB转TTL)

WiFi断连:
  1. 使用AP模式(设备直连)而非STA模式
  2. 固定信道避免自动切换
  3. 改用蓝牙或有线通信

串口通信错误:
  1. 降低波特率(115200→9600)
  2. 加校验(和校验/CRC)
  3. 缩短线缆长度
```

## 第4天：测试与提交阶段

### 应急情况9: 测试现场与实验室环境差异大

**应对方案:**
```
光照不同:
  - 现场快速标定颜色阈值
  - 准备遮光罩/补光灯随时可用
  - 携带黑色背景布

地面不同:
  - 重新标定轮子摩擦系数
  - 调整PID参数(通常现场需要降低响应速度)
  - 准备不同材质的轮子(橡胶轮)

空间不同:
  - 重新校准超声波/激光测距
  - 调整运动范围限制
  - 减速策略更保守
```

### 应急情况10: 最后时刻功能仍有Bug

**应对方案:**
```
1. 功能分层:
   - 必须有的功能(基本要求) → 确保100%稳定
   - 加分功能(发挥部分) → 能跑就行，不稳定也行
   - 锦上添花功能 → 果断放弃

2. 代码策略:
   - 注释掉不稳定的代码段
   - 用简单逻辑替代复杂逻辑
   - 增大安全裕度(宁可慢不要错)

3. 提交前检查:
   - 电源充满/备好备用电池
   - 代码编译通过并烧录确认
   - 从头到尾跑一遍完整流程
   - 打印/导出完整报告
```

## 附录A：硬件应急快速排查手册

### 硬件应急1: MCU烧录失败（SWD/JTAG连接失败）

**现象**：下载器无法识别MCU，提示"No ST-Link detected"或"SWD error"

**快速排查3步法**：
```
步骤1: 物理连接检查 (1分钟)
  - 确认SWD线序: SWDIO→PA13, SWCLK→PA14, GND→GND, VCC→3.3V
  - 检查杜邦线是否松动，更换杜邦线
  - 确认MCU供电正常(3.3V，电流>50mA)

步骤2: 复位后重试 (1分钟)
  - 按住复位键不放，点击下载，释放复位键
  - 或: BOOT0接高电平→复位→进入串口模式→清除Flash→重新烧录
  - 对于STM32F103: PA13/PA14被占用时需手动复位

步骤3: 芯片状态确认 (1分钟)
  - 用万用表测量PA13/PA14对地电压(应为3.3V)
  - 如果为0V: 检查是否被配置为JTAG禁用
  - 如果为3.3V: 尝试更换下载器或MCU
```

**终极方案**：MCU损坏 → 更换备用板，或用串口ISP模式烧录

### 硬件应急2: 电源短路

**现象**：上电后电源指示灯不亮，或电源模块发热严重

**快速定位短路点**：
```
步骤1: 断电，万用表蜂鸣档测VCC-GND阻值 (30秒)
  - 阻值<10Ω → 明确短路
  - 阻值10~100Ω → 可能短路
  - 阻值>1kΩ → 正常

步骤2: 分段排查 (3分钟)
  - 拔掉所有模块，逐个接入，找到引起短路的模块
  - 顺序: MCU→传感器→电机驱动→通信模块
  - 每接入一个模块测一次阻值

步骤3: 定位故障元件 (2分钟)
  - 对短路模块用热风枪逐个拆焊可疑元件
  - 常见短路点: 电解电容反接、MOS管击穿、焊锡桥接
  - 目视检查PCB是否有烧焦痕迹
```

**应急处理**：
- 飞线跳过短路区域（临时供电）
- 用保险丝保护电源（0.5A自恢复保险丝）

### 硬件应急3: 电机不转

**现象**：PWM信号正常但电机不转，或电机抖动不转

**驱动电路故障排查3步法**：
```
步骤1: 独立供电测试 (1分钟)
  - 拔掉电机驱动板，直接用电池(通过开关)给电机供电
  - 电机转 → 驱动电路问题
  - 电机不转 → 电机本体损坏

步骤2: PWM信号确认 (1分钟)
  - 示波器/万用表测量驱动板PWM输入(ENA/ENB)
  - 有PWM(20kHz, 占空比50%) → 驱动板问题
  - 无PWM → MCU输出问题(检查定时器配置)

步骤3: 驱动板输出确认 (1分钟)
  - 测量驱动板电机输出端(OUT1/OUT2)
  - 有电压差(>5V) → 电机接线问题
  - 无电压差 → 驱动芯片损坏(更换L298N/TB6612)
```

**常见原因及修复**：
| 原因 | 现象 | 修复 |
|------|------|------|
| L298N使能脚悬空 | 电机不转 | 将STBY接高电平 |
| TB6612 STBY未使能 | 电机不转 | STBY接3.3V |
| PWM频率错误 | 电机啸叫 | 改为20kHz |
| 电流不足 | 电机堵转 | 增大电池电压/换大电流驱动 |

### 硬件应急4: 传感器无信号

**现象**：ADC读数恒为0或4095，I2C/SPI通信超时

**ADC通信故障排查**：
```
I2C故障排查:
  1. 确认上拉电阻: SDA/SCL各接4.7kΩ上拉到3.3V
  2. 逻辑分析仪查看波形: 是否有ACK响应
  3. 降低I2C速率: 400kHz→100kHz
  4. 检查地址: 用I2C扫描工具确认设备地址

SPI故障排查:
  1. 确认接线: MOSI/MISO/SCK/CS正确连接
  2. 检查时钟极性/相位(CPOL/CPHA)是否匹配
  3. 示波器看SCK是否有输出
  4. 确认CS片选时序正确

ADC故障排查:
  1. 确认参考电压: VREF=3.3V稳定
  2. 检查模拟输入: 用万用表测引脚电压
  3. 降低采样速率: 减少通道数或降低频率
  4. 加滤波电容: 输入端并100nF陶瓷电容
```

### 硬件应急5: OLED不显示

**现象**：OLED屏幕黑屏或显示乱码

**I2C总线问题排查**：
```
步骤1: 电源确认 (30秒)
  - 测量OLED VCC: 应为3.3V(或5V)
  - 测量OLED GND: 确认接地良好

步骤2: I2C通信确认 (1分钟)
  - 逻辑分析仪抓取SDA/SCL波形
  - 正常: 有起始信号+地址字节+ACK
  - 异常: 无波形(MCU未配置)或无ACK(地址错误)

步骤3: 地址确认 (30秒)
  - OLED常见地址: 0x3C 或 0x3D
  - I2C扫描: 0x00~0x7F全扫描确认
  - 多个I2C设备共线时确认地址不冲突

常见修复:
  - SDA/SCL反接 → 交换
  - 缺少上拉 → 加4.7kΩ上拉
  - 地址错误 → 确认0x3C/0x3D
  - I2C速率过高 → 降至100kHz
```

## 附录B：软件应急快速排查手册

### 软件应急1: 程序死机

**现象**：系统无响应，按键/串口无反应，看门狗未复位

**看门狗复位后快速恢复**：
```
步骤1: 确认死机原因 (1分钟)
  - 串口打印最后一条消息(保存到Flash/EEPROM)
  - 检查是否有HardFault Handler输出
  - 记录死机前的操作序列

步骤2: 快速修复 (3分钟)
  - 开启看门狗: IWDG, 超时1秒
  - 添加栈溢出检测: 定期检查MSP
  - 关键数据存Flash: 死机后可恢复

步骤3: 预防措施
  - 所有指针使用前判空
  - 数组访问前检查边界
  - 避免在中断中执行耗时操作
  - 使用volatile关键字保护共享变量
```

**代码示例**：
```c
// 看门狗初始化
void IWDG_Init(void) {
    IWDG->KR = 0x5555;  // 解锁
    IWDG->PR = IWDG_PR_PR_0;  // 分频4, 超时约1秒
    IWDG->RLR = 0xFFF;  // 重装值
    IWDG->KR = 0xCCCC;  // 启动
}

// 主循环喂狗
while(1) {
    IWDG->KR = 0xAAAA;  // 喂狗
    // ... 正常任务
}
```

### 软件应急2: 串口通信丢包

**现象**：串口数据不完整，偶发乱码，数据丢失

**应急方案**：
```
步骤1: 降低波特率 (30秒)
  - 115200bps → 9600bps
  - 简单有效，立即可试

步骤2: 增加校验 (1分钟)
  - 添加和校验: sum(data) & 0xFF
  - 或CRC校验: CRC16/CRC8
  - 接收端校验失败则丢弃该帧

步骤3: 硬件优化 (2分钟)
  - 缩短线缆长度(<1m)
  - 加磁环/屏蔽线
  - 使用差分通信(RS485)替代单端(UART)

步骤4: 软件优化
  - 增加FIFO缓冲区(256字节)
  - DMA接收替代中断接收
  - 帧头帧尾校验(0xAA...0x55)
```

### 软件应急3: PID控制振荡

**现象**：输出值剧烈波动，电机来回摆动，系统不稳定

**快速稳定方法**：
```
紧急止振(30秒):
  1. 大幅降低Kp (减小到原来的1/3)
  2. Kd设为0
  3. Ki设为0
  → 系统变为纯比例控制，响应变慢但稳定

逐步恢复(2分钟):
  1. 缓慢增大Kp，每次+0.5，观察响应
  2. 加入Ki(从0.01开始)，消除稳态误差
  3. 加入Kd(从0.1开始)，抑制超调
  4. 每次调整等待2~3个振荡周期观察效果

终极降级:
  - 如果PID始终调不好 → 用开环+查表法替代
  - 分段PID: 不同区间用不同参数
  - 前馈控制: 补偿已知扰动
```

### 软件应急4: 内存溢出

**现象**：程序跑飞，栈溢出，全局变量被破坏

**RAM不足应急优化**：
```
快速释放RAM (2分钟):
  1. 检查大型数组: 减小数组大小或用全局替代局部
  2. 避免递归: 改用迭代
  3. 使用const数组: 存Flash而非RAM
  4. 关闭printf: 替换为自定义轻量打印

代码优化:
  - char替代int (能用8位不用32位)
  - 位域替代多个bool
  - 共用体(union)复用内存
  - 动态内存改为静态分配

监控手段:
  - 栈指针监控: 定期打印MSP值
  - 堆使用统计: mallinfo()或自定义统计
  - 写入特定值: 0xDEADBEEF检测溢出
```

## 附录C：系统应急快速处理手册

### 系统应急1: 摄像头丢帧

**现象**：图像处理帧率下降，目标检测丢失

**快速处理**：
```
方案1: 降低分辨率 (30秒)
  - 640×480 → 320×240 (帧率×4)
  - 1280×720 → 640×480 (帧率×2)
  - 在OpenCV中: cap.set(CAP_PROP_FRAME_WIDTH, 320)

方案2: 切换USB端口 (30秒)
  - USB2.0 → USB3.0端口
  - 避免USB Hub，直连主板端口
  - 确认USB供电充足(摄像头>500mA)

方案3: 降低处理复杂度 (1分钟)
  - 减少图像预处理步骤
  - 用二值化替代彩色处理
  - 降低检测频率: 30fps→15fps

方案4: 硬件降级
  - 高分辨率摄像头 → 低分辨率摄像头
  - USB摄像头 → OpenMV/K210(内置视觉处理)
```

### 系统应急2: 时间不够（功能裁剪）

**功能裁剪优先级指南**：

```
优先级1 (必须保留):
  □ 基本功能1: 核心控制功能
  □ 基本功能2: 核心传感器功能
  □ 系统稳定性: 不死机、不崩溃

优先级2 (尽量保留):
  □ 基本功能3: 辅助功能
  □ 发挥部分1: 最简单的发挥功能

优先级3 (可裁剪):
  □ 发挥部分2: 复杂度高的发挥功能
  □ 发挥部分3: 需要额外硬件的功能

优先级4 (果断放弃):
  □ 锦上添花功能
  □ 未经验证的新算法
  □ 美观性优化

时间分配建议 (最后24小时):
  6小时: 核心功能联调
  4小时: 基本功能100%稳定
  2小时: 发挥部分尝试
  2小时: 文档整理
  4小时: 休息+最终检查
  6小时: 睡眠(必须!)
```

### 系统应急3: 演示失败

**备用方案设计原则**：
```
原则1: 预录视频备份
  - 提前录好完整演示视频
  - 准备不同环境/条件下的录像
  - 演示失败时可播放视频说明

原则2: 降级演示
  - 自动模式失败 → 手动遥控演示
  - 全功能失败 → 单功能演示
  - 实时处理失败 → 预计算结果展示

原则3: 分步演示
  - 不要一口气演示全部功能
  - 每个功能独立演示
  - 一个功能失败不影响其他

原则4: 现场快速修复
  - 准备最小系统备份板
  - 预烧录好基础功能的固件
  - 5分钟内可切换到备份系统

演示前检查清单:
  □ 电池充满(100%)
  □ 代码编译通过
  □ 从头到尾完整跑一遍
  □ 备份板烧录完成
  □ 视频备份就绪
  □ 报告打印完成
```
