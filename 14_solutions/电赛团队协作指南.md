# 电赛团队协作指南

> 三人团队如何高效协作，在四天三夜中最大化产出

---

## 一、团队角色分工

### 推荐分工方案

| 角色 | 主要职责 | 技能要求 | 人员 |
|------|---------|---------|------|
| **硬件工程师** | 电路设计、焊接、调试 | 模电数电基础、焊接技能 | 1人 |
| **软件工程师** | 程序编写、算法调试 | C语言、单片机编程 | 1人 |
| **系统/报告工程师** | 系统集成、报告撰写、辅助调试 | 文档能力、全局视野 | 1人 |

### 分工注意事项

- **不要绝对割裂**：软件工程师也要能看懂原理图，硬件工程师要能用示波器抓波形
- **系统工程师不是打杂的**：负责整体架构、进度管控、测试验证、报告撰写
- **赛前确定默契**：谁负责哪个模块，接口怎么定义，提前约定好

### 4天3夜时间分配建议

```
Day 1 上午：  选题 + 方案论证（三人一起）
Day 1 下午：  硬件选型搭建 / 软件框架搭建 / 系统工程师开始报告框架
Day 1 晚上：  硬件焊接调试 / 软件核心模块编写
Day 2 全天：  硬件基本完成 / 软件功能逐步实现
Day 3 全天：  系统联调（核心日）
Day 4 上午：  最终调试 + 参数优化
Day 4 下午：  报告完善 + 封装
```

## 二、接口定义与模块化

### 硬件-软件接口约定

在动手之前，三个人必须坐下来确定：

```markdown
## 接口约定表

| 信号 | 引脚 | 方向 | 电平 | 备注 |
|------|------|------|------|------|
| 电机PWM | PA0 | 输出 | 3.3V | TIM2_CH1, 20kHz |
| 编码器A | PA6 | 输入 | 3.3V | TIM3_CH1 |
| 陀螺仪SDA | PB7 | 双向 | 3.3V | I2C1 |
| 显示屏 | SPI1 | 输出 | 3.3V | 128x64 OLED |
| 按键1 | PC0 | 输入 | 上拉 | 启动/停止 |
| ADC电压 | PA1 | 输入 | 0-3.3V | 电池电压检测 |
```

### 软件模块划分

```
main.c          — 系统初始化 + 主循环
motor.c/h       — 电机驱动（PWM、方向）
encoder.c/h     — 编码器读取 + 速度计算
sensor.c/h      — 传感器数据采集（陀螺仪、ADC等）
pid.c/h         — PID 控制算法
display.c/h     — 显示屏驱动 + UI
key.c/h         — 按键处理
protocol.c/h    — 上位机通信协议
```

### 模块间通信规范

```c
// 每个模块对外只暴露 Init() + Get/Set 接口
void Motor_Init(void);
void Motor_SetSpeed(int16_t speed);      // -1000 ~ +1000
int16_t Motor_GetCurrentSpeed(void);

// 内部变量不对外暴露（用 static）
```

## 三、版本管理

### Git 使用规范

即使是电赛这种短期项目，也强烈建议用 Git：

```bash
# 初始化
git init
git add .
git commit -m "初始工程模板"

# 日常提交
git add -A
git commit -m "feat: 添加电机PID控制"
git commit -m "fix: 修复编码器溢出bug"
git commit -m "debug: 调整PID参数 Kp=8.0"
```

### 分支策略（简化版）

```
main        — 稳定可运行的版本
dev         — 日常开发
fix-xxx     — 修复特定问题
```

**黄金规则**：联调前 main 分支必须是能跑的！

### 如果不用 Git

至少做到：
- 每次大改前复制一份工程文件夹，命名为 `项目名_日期_版本`
- 关键节点留备份：`v1_基础功能OK/` `v2_加传感器/` `v3_联调完成/`
- **永远不要只有一个版本**

## 四、沟通技巧

### 日常同步

- **每天至少两次三人会议**：早一次确认当日目标，晚一次汇报进度
- **有问题立刻说**：不要闷头死磕2小时，可能队友一句话就解决了
- **用实物/波形说话**：把示波器屏幕拍照发群里，比描述清楚100倍

### 常见沟通陷阱

| 陷阱 | 表现 | 正确做法 |
|------|------|---------|
| 不好意思说不会 | 硬撑着搞，浪费时间 | 及时求助，分配给更擅长的人 |
| 各干各的不交流 | 软件写好了但硬件接口变了 | 接口变更必须通知所有成员 |
| 甩锅 | "硬件没问题，是软件的bug" | 先一起定位问题，再讨论原因 |
| 完美主义 | 一个功能调到极致 | 时间有限，够用就行，优先完成所有功能 |

### 争吵处理

电赛期间压力大，争吵很正常。原则：
1. **对事不对人**
2. **用数据/现象说话**，不要凭感觉
3. **谁的方案更快能验证就先试谁的**
4. **实在僵持就都试**，花10分钟验证不亏

## 五、联调策略

### 联调原则

1. **逐步集成，不要一次性全接上**
2. **先单独验证每个模块，再组合**
3. **联调顺序**：
   - 电源 → 时钟 → GPIO → 通信 → 传感器 → 执行器 → 闭环控制

### 联调清单

```
□ 电源电压正确（3.3V、5V、12V）
□ 晶振起振
□ 串口能收发数据
□ 传感器能读到数据
□ 电机能正反转
□ 编码器有脉冲
□ 开环控制正常
□ 闭环控制正常
□ 显示屏正常
□ 按键响应正常
□ 整体功能验证
```

### 联调问题定位法

```
问题出现
  → 软件断点/串口打印确认信号到了哪里
  → 示波器确认硬件信号是否正确
  → 缩小范围到具体模块
  → 对比正常工作的参考代码/电路
  → 修复
```

## 七、心态管理

### 保持战斗力

- **轮流休息**：保证每人每天至少睡 4-5 小时
- **备足食物和咖啡**
- **遇到困难时先休息 10 分钟**，效率会更高
- **互相鼓励**：一句话的正向反馈能大幅提升士气

### 时间到了怎么办

- **功能优先于性能**：先跑起来再优化
- **报告不能丢分**：最后 6 小时必须开始写报告
- **测试数据提前收集**：边调边记录，不要最后补
