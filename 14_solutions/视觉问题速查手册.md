# 视觉问题速查手册

> 电赛视觉模块常见问题排查与解决方案

---

## 1. 光照干扰

### 1.1 强光/逆光

| 现象 | 原因 | 解决方案 |
|------|------|----------|
| 画面过曝，白色区域信息丢失 | 强光源直射镜头 | 加遮光罩/调整摄像头角度避开光源 |
| 目标与背景对比度极低 | 逆光导致目标变暗 | 手动降低曝光、增加增益 |
| 色彩识别失效 | 白平衡被强光干扰 | 手动固定白平衡参数 |

### 1.2 弱光/暗光

| 现象 | 原因 | 解决方案 |
|------|------|----------|
| 图像噪声严重，识别率下降 | 环境光不足 | 加LED补光灯（注意用漫射光避免反光） |
| 帧率自动降低 | 长曝光导致 | 手动限制曝光时间下限 |
| 颜色失真 | 自动增益过高 | 固定增益+补光 |

### 1.3 环境光变化

| 现象 | 原因 | 解决方案 |
|------|------|----------|
| 同一算法白天晚上效果不同 | 光照条件变化 | 使用自适应阈值/HSV动态范围 |
| 室内外切换失效 | 色温差异大 | 使用颜色直方图均衡化预处理 |

### 1.4 通用抗光照策略

```
硬件层面:
  - 摄像头加物理遮光罩
  - 使用红外摄像头+红外补光（不受可见光影响）
  - 偏振滤光片减少反光

软件层面:
  - 自适应二值化（OTSU/Canny）
  - HSV颜色空间分离亮度通道
  - 直方图均衡化/CLAHE局部对比度增强
  - 多曝光融合(HDR思路)
```

## 2. 误检/漏检

### 2.1 误检（False Positive）

| 现象 | 原因 | 解决方案 |
|------|------|----------|
| 背景物体被识别为目标 | 特征区分度不够 | 增加形状/面积过滤条件 |
| 噪点被识别为特征 | 二值化阈值不当 | 形态学操作（开运算去噪点） |
| 阴影被误识别 | 阴影与目标颜色接近 | 增加亮度过滤/使用深度信息 |
| 检测框位置不准 | 特征匹配置信度低 | 提高检测阈值、增加NMS |

### 2.2 漏检（False Negative）

| 现象 | 原因 | 解决方案 |
|------|------|----------|
| 目标在视野内但未检测到 | 阈值过高 | 降低检测阈值 |
| 目标部分遮挡时漏检 | 完整性判断过严 | 使用部分匹配/降低面积约束 |
| 运动模糊导致漏检 | 曝光时间内目标移动快 | 缩短曝光时间、提高帧率 |
| 小目标漏检 | 分辨率不足 | 裁剪ROI区域单独处理 |

### 2.3 通用改善策略

```
预处理优化:
  - 高斯模糊去噪 (kernel 3×3 或 5×5)
  - 形态学开运算: 去除小噪点
  - 形态学闭运算: 填充目标内部空洞
  - 连通域分析: 过滤面积过小/过大的区域

特征增强:
  - 多尺度检测（金字塔）
  - 边缘+颜色双重验证
  - 帧间连续性校验（上一帧有、这一帧也应该有）
```

## 3. 掉帧/延迟

### 3.1 帧率低

| 现象 | 原因 | 解决方案 |
|------|------|----------|
| 实际帧率远低于摄像头标称帧率 | USB带宽/处理瓶颈 | 降低分辨率、用MJPEG格式 |
| 处理一帧耗时过长 | 算法复杂度高 | 优化算法/缩小ROI/降低分辨率 |
| 帧率不稳定 | CPU负载波动 | 多线程分离采集和处理 |

### 3.2 延迟高

| 现象 | 原因 | 解决方案 |
|------|------|----------|
| 从采集到结果有明显延迟 | 处理管线太长 | 流水线并行/减少中间步骤 |
| 串口传输延迟 | 波特率低/数据量大 | 提高波特率/压缩数据 |
| 显示延迟 | 图像渲染开销 | 非必要不实时显示/降低显示分辨率 |

### 3.3 优化方案

```
采集优化:
  - 使用USB3.0摄像头/CSI接口摄像头
  - 设置摄像头输出分辨率与处理分辨率一致（避免软件缩放）
  - 使用DMA传输图像数据

处理优化:
  - 裁剪ROI（只处理感兴趣区域）
  - 降分辨率处理（先缩小再识别）
  - 多线程：采集线程 + 处理线程 + 通信线程
  - 使用硬件加速（FPGA预处理/GPU推理）

传输优化:
  - 只传结果坐标，不传图像
  - 使用二进制协议而非文本协议
  - DMA+中断发送
```

## 4. 标定失败

### 4.1 摄像头标定（内参标定）

| 现象 | 原因 | 解决方案 |
|------|------|----------|
| 重投影误差大(>1像素) | 标定图片质量差 | 确保棋盘格平整、图片清晰无模糊 |
| 畸变校正后仍有畸变 | 标定图片数量不足 | 至少15-20张不同角度的图片 |
| 标定矩阵异常 | 棋盘格尺寸设置错误 | 确认棋盘格实际尺寸与代码一致 |
| 标定崩溃 | 图片分辨率不一致 | 所有标定图使用相同分辨率 |

### 4.2 手眼标定

| 现象 | 原因 | 解决方案 |
|------|------|----------|
| 坐标转换误差大 | 标定点分布不均匀 | 标定点覆盖整个工作空间 |
| 结果不稳定 | 标定点数量不足 | 至少10组对应点 |
| 系统性偏差 | 坐标系定义错误 | 画图确认各坐标系方向 |

### 4.3 标定注意事项清单

```
✅ 棋盘格打印在硬板上保证平整
✅ 拍照时棋盘格占画面不同位置（中心、四角、边缘）
✅ 棋盘格倾斜不同角度拍摄
✅ 光照均匀，避免棋盘格上有反光
✅ 图片数量 ≥ 15张
✅ 重投影误差 < 0.5像素为优秀，< 1像素为可接受
✅ 标定后用新图片验证畸变校正效果
```

## 5. 颜色识别问题

### 5.1 HSV颜色空间调参

```
颜色    H范围      S范围      V范围
红色    0-10,      100-255,   100-255
        170-180
黄色    15-35,     100-255,   100-255
绿色    35-85,     100-255,   100-255
蓝色    100-130,   100-255,   100-255

提示: OpenCV中H范围是0-180(非0-360), S和V范围是0-255
```

### 5.2 常见颜色识别问题

| 问题 | 解决方案 |
|------|----------|
| 红色在HSV中跨越0°边界 | 分两段检测后合并：[0,10]和[170,180] |
| 颜色在不同光照下偏移 | 扩大S和V的范围下限，或使用自适应阈值 |
| 白色/黑色难以区分 | 主要靠V通道判断，配合S接近0 |
| 颜色相近区分困难 | 使用LAB颜色空间，或增加S通道约束 |

## 6. OpenCV/视觉库常见坑

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| cv::imread读取空图像 | 路径含中文/特殊字符 | 使用绝对路径或imdecode |
| 找不到轮廓 | 二值化图像类型错误 | 确保输入是CV_8UC1 |
| findContours后内存暴涨 | 未释放vector | 及时clear/swap |
| 模板匹配精度低 | 模板大小/旋转变化 | 使用多尺度匹配或特征点匹配 |
| 视频打不开 | 编解码器缺失 | 安装ffmpeg/使用V4L2 |

### 6.1 实战案例

**案例1：光照变化导致颜色检测失败的处理**

*问题现象*：2025年E题瞄准系统，在不同光照下颜色检测失效，靶心识别率从95%降至30%。

*问题分析*：
```
光照变化影响:
  - 日光灯下: HSV颜色偏暖（H值偏小）
  - LED补光: 颜色饱和度变化（S值波动）
  - 自然光: 亮度渐变（V值大范围变化）
```

*解决方案*：
```python
# 自适应颜色检测
class AdaptiveColorDetector:
    def __init__(self):
        self.reference_hsv = None
        self.h_margin = 10
        self.s_margin = 30
        self.v_margin = 40

    def calibrate_reference(self, frame, roi):
        """标定参考颜色"""
        hsv = cv2.cvtColor(frame, cv2.COLOR_BGR2HSV)
        roi_hsv = hsv[roi[1]:roi[3], roi[0]:roi[2]]
        self.reference_hsv = np.mean(roi_hsv.reshape(-1, 3), axis=0)
        print(f"标定颜色: H={self.reference_hsv[0]:.0f}, S={self.reference_hsv[1]:.0f}, V={self.reference_hsv[2]:.0f}")

    def detect_with_tolerance(self, frame):
        """容差检测"""
        hsv = cv2.cvtColor(frame, cv2.COLOR_BGR2HSV)
        lower = np.array([
            max(0, self.reference_hsv[0] - self.h_margin),
            max(0, self.reference_hsv[1] - self.s_margin),
            max(0, self.reference_hsv[2] - self.v_margin)
        ])
        upper = np.array([
            min(179, self.reference_hsv[0] + self.h_margin),
            min(255, self.reference_hsv[1] + self.s_margin),
            min(255, self.reference_hsv[2] + self.v_margin)
        ])
        mask = cv2.inRange(hsv, lower, upper)
        return mask

    def adaptive_threshold(self, frame):
        """根据亮度自适应调整阈值"""
        gray = cv2.cvtColor(frame, cv2.COLOR_BGR2GRAY)
        brightness = np.mean(gray)

        # 亮度越低，放宽阈值
        factor = 1.0 + (128 - brightness) / 200.0
        self.s_margin = int(30 * factor)
        self.v_margin = int(40 * factor)
```

*效果*：光照变化时识别率保持85%以上。

**案例2：摄像头帧率不足的优化方案**

*问题现象*：2023年E题运动目标追踪，双摄像头同时工作，实际帧率仅25fps（要求30fps）。

*问题分析*：
```
原始性能分析:
  - USB3.0摄像头: 标称120fps
  - OpenCV采集+格式转换: 15ms
  - 图像处理: 20ms
  - 总延迟: 35ms = 28fps
```

*优化步骤*：
```python
# 优化1: 减少格式转换
class OptimizedCamera:
    def __init__(self, device):
        self.cap = cv2.VideoCapture(device)
        self.cap.set(cv2.CAP_PROP_FOURCC, cv2.VideoWriter_fourcc('M', 'J', 'P', 'G'))
        # MJPEG格式比YUYV快2-3倍

    def get_frame_optimized(self):
        ret, frame = self.cap.read()
        if not ret:
            return None
        # 直接使用灰度（避免颜色转换）
        gray = cv2.cvtColor(frame, cv2.COLOR_BGR2GRAY)
        return gray  # 节省5ms

# 优化2: ROI裁剪
def process_roi(frame, roi_ratio=0.5):
    h, w = frame.shape[:2]
    roi = frame[int(h*roi_ratio):, :]  # 只处理下半部分
    return roi  # 处理面积减少50%

# 优化3: 降低分辨率
def dual_camera_optimized():
    cam1 = cv2.VideoCapture(0)
    cam1.set(cv2.CAP_PROP_FRAME_WIDTH, 320)  # 640→320
    cam1.set(cv2.CAP_PROP_FRAME_HEIGHT, 240)  # 480→240
    # 处理速度提升4倍
```

*效果*：帧率从25fps提升至45fps，满足要求。

**案例3：OpenCV内存泄漏的排查方法**

*问题现象*：连续运行30分钟后，Python进程内存占用从200MB增至2GB，最终OOM崩溃。

*排查流程*：
```python
# 1. 定位泄漏来源
import tracemalloc
tracemalloc.start()

# 在循环中定期检查
for i in range(1000):
    frame = process_frame()
    if i % 100 == 0:
        current, peak = tracemalloc.get_traced_memory()
        print(f"内存: {current/1024/1024:.1f}MB, 峰值: {peak/1024/1024:.1f}MB")

# 2. 发现未释放的OpenCV Mat
# 错误代码:
def process_frame(frame):
    gray = cv2.cvtColor(frame, cv2.COLOR_BGR2GRAY)
    blurred = cv2.GaussianBlur(gray, (5, 5), 0)
    _, binary = cv2.threshold(blurred, 128, 255, cv2.THRESH_BINARY)
    return binary  # 中间变量gray, blurred未释放！

# 正确代码:
def process_frame_fixed(frame):
    gray = cv2.cvtColor(frame, cv2.COLOR_BGR2GRAY)
    blurred = cv2.GaussianBlur(gray, (5, 5), 0)
    _, binary = cv2.threshold(blurred, 128, 255, cv2.THRESH_BINARY)
    # 方法1: 显式释放
    del gray, blurred
    # 方法2: 使用with语句（C++端）
    return binary

# 3. 监控内存使用
import psutil
def monitor_memory():
    process = psutil.Process()
    mem_mb = process.memory_info().rss / 1024 / 1024
    if mem_mb > 1000:  # 超过1GB
        print(f"警告: 内存使用{mem_mb:.1f}MB")
        # 手动GC
        import gc
        gc.collect()
```

*根本原因*：OpenCV的`cvtColor`等函数在Python中创建临时Mat，Python垃圾回收不及时。

*解决方案*：
```python
# 方案1: 循环中定期GC
import gc
gc.collect()  # 每100帧调用一次

# 方案2: 复用预分配的Mat
class FrameProcessor:
    def __init__(self):
        self.buffer1 = np.zeros((480, 640), dtype=np.uint8)
        self.buffer2 = np.zeros((480, 640), dtype=np.uint8)

    def process(self, frame):
        # 使用预分配缓冲区
        cv2.cvtColor(frame, cv2.COLOR_BGR2GRAY, dst=self.buffer1)
        cv2.GaussianBlur(self.buffer1, (5, 5), 0, dst=self.buffer2)
        return self.buffer2  # 无内存泄漏
```

*效果*：内存使用稳定在200-250MB，长时间运行无泄漏。

## 7. 嵌入式视觉优化（无OpenCV）

```
常用替代方案:
  - OpenMV: 自带色块识别、二维码识别
  - K210/STM32H7: 使用轻量化神经网络
  - FPGA: 硬件流水线实现阈值化、连通域标记
  - 自写算法: 简单二值化+连通域+面积过滤

嵌入式视觉必须优化:
  - 降分辨率（320×240通常够用）
  - 只处理ROI区域
  - 定点数替代浮点数
  - 使用查表法替代实时计算
```

## 8. 视觉问题快速排查清单

- [ ] 摄像头是否正确连接（USB/CSI接口确认）
- [ ] 图像是否正常采集（先保存一帧图片确认）
- [ ] 光照条件是否满足（关闭干扰光源/补充光源）
- [ ] 颜色阈值是否合适（用滑条实时调参）
- [ ] ROI是否设置正确（确认检测区域）
- [ ] 帧率是否满足需求（查看实际FPS）
- [ ] 处理结果坐标系是否与控制坐标系一致
