# 电赛4天3夜完整攻略 - 时间管理·任务分解·团队协作

## 一、赛事时间线

| 时间 | 阶段 | 核心任务 |
|------|------|----------|
| D1 8:00 | 开赛 | 选题、分析、方案论证 |
| D1 8:00-12:00 | 方案确定 | 确定方案、器件清单、分工 |
| D1 12:00-20:00 | 硬件搭建 | PCB/面包板、焊接、基本电路 |
| D1 20:00-24:00 | 模块调试 | 各模块独立测试通过 |
| D2 全天 | 核心开发 | 主功能实现、联调 |
| D3 全天 | 系统联调 | 完整系统运行、优化参数 |
| D3 晚上 | 测试完善 | 边界测试、稳定性验证 |
| D4 8:00-12:00 | 报告撰写 | 设计报告、数据整理 |
| D4 12:00-15:00 | 最终验收 | 测试演示 |
| D4 15:00 | 提交 | 报告+实物 |

## 二、Day 1 详细计划

### 8:00-9:00 - 选题分析（全队）
- [ ] 仔细读题3遍，标注关键词
- [ ] 列出所有指标（基本/发挥）
- [ ] 评估难度，确定做哪些发挥部分
- [ ] **决策点：选择哪道题（A/B/C/D类）**

### 9:00-10:00 - 方案论证（全队讨论）
- [ ] 头脑风暴所有可行方案
- [ ] 评估每个方案：可行性、工作量、风险
- [ ] 确定主方案+备选方案
- [ ] 画出系统框图

### 10:00-12:00 - 详细设计与分工
- [ ] 硬件架构设计（画原理图草图）
- [ ] 软件架构设计（画流程图）
- [ ] 列出器件清单（确认手头有无）
- [ ] **三个人明确分工**

### 团队分工模板（3人队）

| 角色 | Day1 | Day2 | Day3 | Day4 |
|------|------|------|------|------|
| **硬件** | 电路搭建、焊接 | 驱动调试 | 硬件优化 | 辅助测试 |
| **软件-核心** | 底层驱动编写 | 核心算法 | 算法优化 | 代码整理 |
| **软件-上层** | 通信/UI框架 | 功能集成 | 系统联调 | 写报告 |

### 12:00-20:00 - 硬件搭建
- [ ] 确认所有器件到手
- [ ] 画PCB/面包板布局
- [ ] 焊接核心板
- [ ] 电源电路先行测试
- [ ] 逐个模块焊接测试

### 20:00-24:00 - 模块调试
- [ ] 每个硬件模块独立通电测试
- [ ] 软件跑通底层驱动（GPIO/UART/SPI/I2C/ADC/PWM）
- [ ] 串口打印确认数据正确
- [ ] **当日结束前必须确认：所有硬件模块单独可用**

### ⚠️ Day1 避坑
- 不要在方案上纠结太久（2小时内必须定方案）
- 硬件先行，软件并行写框架
- 不确定的器件先买两套方案的
- 电源一定要稳，先测电源再接其他

## 三、Day 2 详细计划

### 核心原则：**功能优先，完善其次**

### 上午（8:00-12:00）
- [ ] 软件：实现最基本的功能闭环
  - 例：电机能转、传感器能读、能通信
- [ ] 硬件：辅助软件调试，修bug
- [ ] **里程碑：最小可用系统跑通**

### 下午（12:00-20:00）
- [ ] 实现核心功能模块
  - 例：自动控制算法、信号处理、测量算法
- [ ] 开始初步联调
- [ ] 记录问题清单

### 晚上（20:00-24:00）
- [ ] 继续联调，修复发现的问题
- [ ] 整理代码，加注释
- [ ] **里程碑：基本功能可用（能拿到基础分）**

### ⚠️ Day2 避坑
- 不要追求完美，先跑通再说
- 遇到硬件问题先怀疑接线，再怀疑代码
- 多用串口调试，不要盲猜
- 定时吃饭休息，保持效率

## 四、Day 3 详细计划

### 核心原则：**优化+稳定+完善**

### 上午（8:00-12:00）
- [ ] 实现发挥部分功能
- [ ] 优化核心算法参数（PID调参等）
- [ ] 测试各边界条件

### 下午（12:00-20:00）
- [ ] 完整系统联调
- [ ] 反复测试稳定性（跑10次看成功率）
- [ ] 修bug、优化性能
- [ ] 开始整理报告素材（截图、数据）

### 晚上（20:00-24:00）
- [ ] 最终测试：模拟验收流程
- [ ] 准备演示流程
- [ ] 拍照/录像留档
- [ ] **里程碑：系统稳定运行，可演示**

### ⚠️ Day3 避坑
- 不要在Day3加新功能，只优化已有功能
- 一定要做稳定性测试
- 准备好备用方案（某功能不行就放弃）
- 保存好所有代码版本

## 五、Day 4 上午

### 报告写作（4小时内完成）

```
报告结构（约15-20页）：
1. 摘要（1页）- 方案概述、核心指标、测试结果
2. 方案论证（2页）- 方案比较、系统框图
3. 理论分析（2页）- 关键公式推导
4. 电路设计（3页）- 原理图、关键参数计算
5. 程序设计（3页）- 流程图、关键代码
6. 测试结果（3页）- 数据表格、波形图
7. 总结（1页）
```

### 最终验收准备
- [ ] 检查所有接线
- [ ] 确认电源电量充足
- [ ] 准备好测试步骤（先演示什么再演示什么）
- [ ] 确认报告已提交

## 六、团队协作规范

### 沟通
- **每2小时同步一次进度**（5分钟站立会）
- 遇到阻塞问题立即沟通，不要闷头死磕
- 使用共享文档记录问题和决策

### 代码管理
- 用Git管理代码（或至少U盘备份）
- 主分支保持可编译运行
- 每个功能模块独立文件
- 共用头文件统一接口定义

### 冲突解决
- 技术分歧：**快速实验验证**，不要空谈
- 时间冲突：**进度优先**，先实现再优化
- 疲劳决策：**休息后重审**，避免低级错误

## 七、应急策略

### 场景1：硬件烧了
→ 备用器件在哪？是否有替代方案？

### 场景2：算法调不通
→ 降级方案是什么？放弃发挥部分保住基础分

### 场景3：时间不够
→ 砍功能清单，按优先级排序：
1. 基本功能（必须）
2. 发挥部分A（重要）
3. 发挥部分B（锦上添花）
4. 完美优化（来不及就放弃）

### 场景4：队员体力不支
→ 轮流休息，至少保证1人清醒值班
→ Day2凌晨建议至少睡3-4小时
