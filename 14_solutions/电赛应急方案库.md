# 电赛应急方案库

> 赛场上常见突发情况及应对策略，有备无患

---

## 一、硬件故障应急

### 1.1 MCU 烧毁

**现象**：芯片发烫、不工作、程序无法下载

**应急方案**：
- **赛前必带备用板**：至少准备 2 块同型号开发板
- 如果没有备用板，检查是否只是供电部分损坏，尝试外部供电
- 从备用板飞线核心引脚到现有电路
- **预防**：接线前用万用表确认电源和地，不要带电插拔

### 1.2 电源故障

**现象**：系统不工作、电压异常、有烧焦味

**应急排查流程**：

```
1. 断开所有负载，测量电源输出
2. 如果电源正常 → 逐个接回负载，找出短路模块
3. 如果电源异常 → 检查电源芯片、滤波电容
4. 使用备用电源模块替换
```

**应急方案**：
- 准备多个 DC-DC 降压模块（MP1584/LM2596）
- 准备 LDO 备用（AMS1117-3.3/5.0）
- 用电池 + 降压模块替代不可靠的桌面电源
- **赛前检查**：电源空载和满载电压都测一下

### 1.3 电机/舵机故障

**现象**：不转、抖动、异响、冒烟

| 故障现象 | 可能原因 | 应急处理 |
|---------|---------|---------|
| 完全不转 | 驱动芯片烧毁 | 更换驱动模块（L298N/TB6612） |
| 一方向不转 | H桥一路损坏 | 更换驱动或飞线到另一路 |
| 剧烈抖动 | PWM频率太低 | 提高频率到20kHz以上 |
| 舵机抖动 | 供电不足 | 独立供电，加大电容 |
| 发热严重 | 堵转/过载 | 检查机械卡死，降低电流 |

### 1.4 传感器数据异常

**现象**：数据跳变、卡死、全零/满量程

**排查步骤**：
1. 检查接线（特别是电源和地）
2. 用示波器/逻辑分析仪抓通信波形
3. 检查 I2C/SPI 地址是否正确
4. 检查上拉电阻（I2C 需要 4.7K 上拉）
5. 降低通信速率试试
6. **最后手段**：换一个传感器模块

### 1.5 显示屏不亮

**现象**：OLED/LCD 无显示

**排查**：
- 检查对比度/背光设置
- 检查 I2C/SPI 通信（逻辑分析仪抓波形）
- 确认屏幕地址（OLED 常见 0x3C 或 0x3D）
- 检查初始化代码是否正确执行

## 二、软件故障应急

### 2.1 程序卡死/死循环

**现象**：系统无响应，按键无效

**应急方案**：
- 按复位键
- 检查是否有死循环或阻塞式等待
- 在主循环加 LED 闪烁指示（看门狗）
- 串口打印关键变量定位卡死位置

```c
// 加看门狗保护（赛前配置好）
// STM32 独立看门狗
IWDG_WriteAccessCmd(IWDG_WriteAccess_Enable);
IWDG_SetPrescaler(IWDG_Prescaler_256);
IWDG_SetReload(156);  // 约1秒超时
IWDG_ReloadCounter();
IWDG_Enable();

// 主循环中定期喂狗
while(1) {
    IWDG_ReloadCounter();  // 喂狗
    // ... 正常任务
}
```

### 2.2 通信故障

**现象**：串口/I2C/SPI 通信失败

| 故障 | 排查方法 |
|------|---------|
| 串口乱码 | 检查波特率是否一致 |
| I2C 无应答 | 检查地址、上拉电阻、总线是否被拉低 |
| SPI 数据错 | 检查时钟极性和相位（CPOL/CPHA） |
| 通信时好时坏 | 检查接线是否松动，加短接线 |

### 2.3 程序空间不足

**现象**：编译报错"region FLASH overflowed"

**应急方案**：
- 优化等级设为 -O2 或 -Os
- 删除未使用的函数和库
- 减少 printf 使用（占用大量空间）
- 使用 MicroLib（Keil 选项）
- 去掉不需要的断言 assert

### 2.4 浮点运算太慢

**现象**：控制周期不稳定，PID 运算超时

**应急方案**：
- 使用定点数运算（放大100倍用整数）
- 减少三角函数调用（查表法）
- 简化算法（PID 用整数版本）
- 提高系统时钟频率

## 三、联调故障应急

### 3.1 接口不匹配

**现象**：模块间信号电平不兼容、引脚对不上

**应急方案**：
- 3.3V → 5V：用 74HC245 或三极管电平转换
- 5V → 3.3V：大部分 3.3V MCU 5V 容忍（确认数据手册）
- 引脚不够：用移位寄存器（74HC595）扩展输出
- 引脚对不上：飞线！杜邦线 + 飞线是万能的

### 3.2 系统不稳定（时好时坏）

**可能原因及解决**：

| 原因 | 解决方案 |
|------|---------|
| 电源纹波大 | 加 100μF + 100nF 滤波电容 |
| 信号串扰 | 模拟信号线远离数字信号线 |
| 接地不良 | 所有模块共地，地线要粗 |
| 接线松动 | 焊死关键连线，不要只靠杜邦线 |
| 电磁干扰 | 电机驱动板远离传感器和MCU |

### 3.3 功能联调顺序

如果时间紧迫，按优先级联调：

```
第一优先级（必须完成）：
  → 电源稳定
  → 核心功能（题目基本要求）

第二优先级（尽量完成）：
  → 发挥部分基础功能
  → 传感器数据正常

第三优先级（有时间再做）：
  → 发挥部分进阶功能
  → 性能优化
  → 显示美化
```

## 四、比赛日应急

### 4.1 检查时设备被收走

**预防**：提前确认赛场允许的工具和设备清单
**应急**：准备替代方案（如示波器被收走就用串口打印波形数据）

### 4.2 最后时刻功能还没调好

**策略**：
1. 确保基本功能稳定运行（先拿基础分）
2. 报告先写好（报告分不能丢）
3. 发挥部分能做多少做多少
4. **不要在最后2小时做大改动**，稳定比功能多更重要

### 4.3 队友身体不适

**预防**：备好常用药（感冒药、止痛药、肠胃药）
**应急**：
- 轻微不适：休息 + 吃药，另外两人顶上
- 严重不适：先就医，两人继续推进

### 4.4 器材损坏且无备用

**应急思路**：
- 问隔壁队伍借（电赛大家互帮互助）
- 用其他模块替代（如 L298N 坏了用 MOS 管搭 H 桥）
- 简化方案（去掉非核心功能）
- 如果关键模块无替代，调整方案实现路径

## 五、预防措施清单

### 赛前准备

```
硬件备份：
□ 同型号开发板 × 2
□ 电机驱动模块 × 2
□ 传感器模块 × 2（常用型号）
□ 电源模块 × 2
□ USB 数据线 × 3
□ ST-Link × 2

软件备份：
□ 工程文件 U盘备份
□ 云盘同步
□ 基础代码模板（HAL库、驱动代码）
□ 参考代码库（历届优秀作品代码）

生活保障：
□ 充足的食物和水
□ 提神饮料
□ 常用药品
□ 毯子/睡袋
```

### 赛中纪律

- **每 2 小时保存一次工程**（Git commit / 文件备份）
- **修改前先备份当前可运行版本**
- **调试时先开串口，再上电电机**
- **焊接前断电，焊接后检查短路再通电**
- **更换元件前测量确认型号和引脚**
