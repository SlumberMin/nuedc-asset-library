## 高分报告模板

**文件**: `高分报告模板.md`

### 模板结构

| 章节 | 内容 | 建议页数 |
|------|------|---------|
| 一、摘要 | 系统概述+核心技术亮点+关键指标 | 0.5页 |
| 二、系统方案论证与选择 | 方案对比+选择理由+系统框图 | 2~3页 |
| 三、理论分析与计算 | 关键公式推导+参数计算 | 2~3页 |
| 四、电路与程序设计 | 硬件电路+软件流程 | 3~4页 |
| 五、测试方案与数据 | 测试方法+测试数据+结果分析 | 2~3页 |
| 六、总结 | 创新点+不足+改进方向 | 0.5页 |

> **总页数建议**: 12-15页（不含代码附录），正文行距1.5倍。

### 摘要模板

```markdown
## 一、摘要

### 1.1 系统概述

### 1.2 核心技术亮点

### 1.3 关键指标达成

[指标1：实测值 ≥ 要求值]，[指标2：实测值 ≤ 要求值]。
```

## 报告自动生成脚本

**文件**: `报告自动生成脚本.py`

### 功能概述

通过交互式输入或配置文件，快速生成完整的 Markdown 格式设计报告框架。

### 使用方法

```bash
# 交互式模式 — 逐项询问填写
python 报告自动生成脚本.py

# 配置文件模式 — 从JSON读取配置
python 报告自动生成脚本.py -c config.json

# 演示模式 — 生成演示报告
python 报告自动生成脚本.py --demo
```

### 脚本API

#### `ReportGenerator` 类

```python
from 报告自动生成脚本 import ReportGenerator, DEFAULT_CONFIG

# 使用默认配置
gen = ReportGenerator()

# 使用自定义配置
config = {**DEFAULT_CONFIG, "title": "我的报告"}
gen = ReportGenerator(config=config)

# 从文件加载
gen = ReportGenerator(config_path="config.json")
```

| 方法 | 参数 | 返回 | 说明 |
|------|------|------|------|
| `generate()` | - | str | 生成完整Markdown报告 |
| `save(path)` | 输出路径 | - | 保存报告到文件 |
| `generate_abstract()` | - | str | 仅生成摘要部分 |
| `generate_comparison_table(module)` | 模块配置 | str | 生成方案对比表 |
| `generate_test_table(specs)` | 指标列表 | str | 生成测试数据表 |

#### 使用示例

```python
from 报告自动生成脚本 import ReportGenerator

gen = ReportGenerator(config_path="my_config.json")
report = gen.generate()
gen.save("设计报告.md")
print("报告已生成: 设计报告.md")
```

## 报告写作指南

**文件**: `电赛设计报告写作指南.md`

### 写作原则

| 原则 | 说明 |
|------|------|
| 结构清晰 | 严格按模板章节组织，逻辑连贯 |
| 图文并茂 | 每个关键模块配框图/原理图/实物图 |
| 数据说话 | 测试数据用表格呈现，标注单位和条件 |
| 对比论证 | 方案选择必须有对比，说明选择理由 |
| 突出创新 | 摘要和总结突出技术创新点 |

### 各章节写作要点

#### 摘要

- **第一段**: 系统概述（2-3句），说清楚用什么、做什么
- **第二段**: 核心技术亮点（3-5句），硬件+软件+创新点
- **第三段**: 关键指标达成（1-2句），量化结果
- **总字数**: 200-300字，不超过半页

#### 方案论证

- 每个关键模块都要有方案对比（至少2个方案）
- 用**表格**对比优缺点
- 给出明确的**选择理由**
- 画**系统总体框图**

#### 理论分析

- 关键公式推导要**完整**
- 参数计算要**有依据**
- 标注**参考文献或标准**

#### 电路设计

- 每个模块画**电路原理图**
- 标注**关键参数**（电阻值、电容值、增益等）
- 说明**设计依据**（为什么选这个参数）

#### 程序设计

- 画**主程序流程图**
- 关键算法画**子流程图**
- 核心代码加**注释**（不要贴大段代码）

#### 测试方案

- 明确**测试条件**（环境、仪器、方法）
- 用**表格**呈现测试数据
- 数据要**完整**（包含所有要求的测试项）
- 最后一列写**是否达标**

### 语言规范

| 要求 | 示例 |
|------|------|
| 用词准确 | "采用"而非"使用了" |
| 句式简洁 | "系统由XX模块组成" |
| 避免口语化 | "经测试"而非"我们测了一下" |
| 数据精确 | "精度为0.5%"而非"精度很高" |

## 报告评分要点

### 评分维度

| 维度 | 权重 | 关注点 |
|------|------|--------|
| 方案论证 | 20% | 方案对比是否充分、选择理由是否合理 |
| 理论分析 | 20% | 公式推导是否正确、参数计算是否有据 |
| 电路设计 | 20% | 电路是否正确、参数是否合理 |
| 程序设计 | 15% | 流程是否清晰、算法是否合理 |
| 测试方案 | 15% | 测试方法是否科学、数据是否完整 |
| 报告规范 | 10% | 格式、排版、语言表达 |

### 加分项

- 创新点突出
- 测试数据超出题目要求
- 图表精美、排版专业
- 有对比实验数据

### 扣分项

- 方案无对比、无理由
- 公式推导缺失
- 电路图缺失或不完整
- 测试数据不全
- 格式混乱、错别字

## 常见问题与注意事项

### 常见问题

| 问题 | 解决方案 |
|------|---------|
| 报告写不完 | 先写框架，再填内容；用自动生成脚本 |
| 方案没有对比 | 参考模板中的对比表格式，至少列2个方案 |
| 测试数据不全 | 提前规划测试项目，逐项测试记录 |
| 图片不清楚 | 截图时选择高分辨率，必要时重新绘制 |
| 排版混乱 | 使用Markdown统一格式，导出PDF前检查 |

### 时间分配建议

```
Day 1 上午: 读题 + 方案确定
Day 1 下午: 开始硬件搭建 + 撰写方案论证章节
Day 2: 硬件调试 + 撰写电路/程序设计章节
Day 3 上午: 系统测试 + 记录测试数据
Day 3 下午: 撰写测试方案 + 摘要 + 总结 + 排版
```

### 报告检查清单

- [ ] 摘要是否包含系统概述、技术亮点、指标达成
- [ ] 每个模块是否有方案对比表
- [ ] 系统框图是否完整
- [ ] 关键公式是否推导完整
- [ ] 电路原理图是否标注关键参数
- [ ] 程序流程图是否清晰
- [ ] 测试条件是否明确
- [ ] 测试数据是否用表格呈现
- [ ] 测试项目是否覆盖所有题目要求
- [ ] 格式排版是否统一
- [ ] 页数是否在12-15页范围内
