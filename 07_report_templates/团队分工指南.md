# 电赛三人团队分工与协作指南

---

## 一、角色定义与职责

### 角色A：硬件工程师（队长建议人选）

**核心职责：**
- 电路设计与PCB绘制
- 焊接与硬件组装
- 电源系统设计
- 传感器选型与接线
- 机械结构搭建与调试

**赛前准备：**
- 掌握至少一款PCB设计工具（KiCad/立创EDA/AD）
- 准备常用的电机驱动模块原理图
- 熟悉各种传感器的接线和调试
- 准备常用器件清单和备件

**比赛期间工作：**
| 时间段 | 工作内容 |
|--------|---------|
| Day1上午 | 参与选题讨论，评估硬件可行性 |
| Day1下午 | 搭建硬件平台，完成基本接线 |
| Day1晚上 | 硬件联调，排除硬件故障 |
| Day2 | 硬件优化，PCB打样或洞洞板搭建 |
| Day3 | 硬件加固、可靠性测试 |
| Day4 | 最终检查，协助测试 |

---

### 角色B：软件工程师

**核心职责：**
- 单片机程序开发
- 算法实现（PID、滤波、路径规划等）
- 传感器驱动编写
- 通信协议实现
- 上位机开发（如需要）

**赛前准备：**
- 精通STM32开发
- 准备成熟的驱动库（电机、传感器、通信）
- 掌握PID调参技巧
- 准备常用的算法模板

**比赛期间工作：**
| 时间段 | 工作内容 |
|--------|---------|
| Day1上午 | 参与选题讨论，评估软件可行性 |
| Day1下午 | 搭建软件框架，编写基础驱动 |
| Day1晚上 | 核心算法实现与调试 |
| Day2 | 功能实现，与硬件联调 |
| Day3 | 算法优化，性能提升 |
| Day4 | 最终调试，bug修复 |

---

### 角色C：报告工程师 + 测试

**核心职责：**
- 报告撰写与排版
- 测试方案设计与执行
- 数据记录与分析
- 演示视频录制
- 答辩准备

**赛前准备：**
- 准备报告模板（参考本资产库）
- 练习技术报告写作
- 了解评分标准
- 准备测试仪器使用技能

**比赛期间工作：**
| 时间段 | 工作内容 |
|--------|---------|
| Day1上午 | 记录选题讨论过程，开始写方案论证 |
| Day1下午 | 继续报告，同时协助硬件搭建 |
| Day1晚上 | 记录调试过程和数据 |
| Day2 | 撰写理论分析，设计测试方案 |
| Day3 | 执行测试，记录数据，撰写测试部分 |
| Day4 | 报告定稿，排版，打印 |

## 二、协作规范

### 2.1 沟通机制
- **每日三次碰头**：早/中/晚各10分钟，同步进度
- **问题即时沟通**：遇到阻塞问题立即告知队友
- **决策机制**：方案分歧时队长做最终决定

### 2.2 代码管理
```
项目结构/
├── Hardware/          # 硬件相关
│   ├── Schematic/     # 原理图
│   └── PCB/           # PCB文件
├── Software/          # 软件相关
│   ├── Drivers/       # 驱动层
│   ├── App/           # 应用层
│   ├── Algorithm/     # 算法层
│   └── Config/        # 配置文件
├── Docs/              # 文档
│   ├── Report/        # 报告
│   └── Datasheet/     # 器件手册
└── Test/              # 测试记录
```

### 2.3 接口约定
- **硬件-软件接口**：提前约定引脚分配，形成文档
- **模块间通信**：统一串口协议格式
- **命名规范**：统一变量和函数命名规则

```
// 推荐的引脚分配表模板
| 功能 | 引脚 | 方向 | 说明 |
|------|------|------|------|
| 电机A_PWM | PA0 | OUT | TIM2_CH1 |
| 电机A_DIR | PA1 | OUT | 正反转 |
| 编码器A | PA6/PA7 | IN | TIM3_CH1/CH2 |
| 超声波_Trig | PB0 | OUT | 触发信号 |
| 超声波_Echo | PB1 | IN | 回波信号 |
| MPU_SDA | PB7 | I2C | I2C1_SDA |
| MPU_SCL | PB6 | I2C | I2C1_SCL |
| 蓝牙_TX | PA9 | UART | USART1_TX |
| 蓝牙_RX | PA10 | UART | USART1_RX |
```

### 2.4 文件同步
- 使用U盘/网盘实时同步
- 代码使用Git管理（即使本地也用）
- 每次修改前备份关键版本

## 三、协作流程图

```
选题阶段（三人共同）
    │
    ├── 硬件可行性评估 ← 硬件负责
    ├── 软件可行性评估 ← 软件负责
    └── 评分标准分析 ← 报告负责
    │
    ▼
方案确定（三人讨论）
    │
    ├── 硬件方案设计 → 硬件执行
    ├── 软件架构设计 → 软件执行
    └── 报告大纲 → 报告执行
    │
    ▼
并行开发
    │
    ├── 硬件搭建 ──→ 硬件联调（硬件+软件）
    ├── 软件开发 ──┘
    └── 报告撰写（报告持续进行）
    │
    ▼
联调测试（三人共同）
    │
    ├── 功能测试 ← 报告记录
    ├── 性能测试 ← 报告记录
    └── 问题修复 ← 硬件/软件
    │
    ▼
收尾阶段
    │
    ├── 报告定稿 ← 报告主责
    ├── 作品加固 ← 硬件主责
    └── 最终调试 ← 软件主责
```

## 四、常见协作问题与解决方案

### 问题1：硬件还没好，软件没东西调
**解决**：软件先用仿真/模拟数据调试算法，准备好驱动代码

### 问题2：调试时互相干扰
**解决**：明确分工区域，调试时一人操作一人记录

### 问题3：报告来不及写
**解决**：报告工程师从Day1就开始写，不要等到最后

### 问题4：选题意见不统一
**解决**：投票决定，2:1即通过；都不确定时选简单的

### 问题5：队友能力差距大
**解决**：强者多承担核心任务，弱者负责辅助和报告

## 五、赛前团队建设

### 5.1 模拟赛（至少2次）
- 选历年真题，4天3夜完整练习
- 按正式比赛标准要求
- 赛后复盘，找出薄弱环节

### 5.2 技能互补培训
- 硬件教软件基本焊接和接线
- 软件教硬件基本编程和调试
- 报告教大家测试方法和数据记录

### 5.3 默契培养
- 建立常用的简短沟通暗号
- 统一调试习惯（如LED指示灯含义）
- 约定紧急情况的处理流程

## 八、并行工作流程设计

### 8.1 关键并行点
```
Day1 并行:
  硬件: 原理图绘制 ←→ 软件: 工程配置
  硬件: 核心电路焊接 ←→ 软件: 驱动代码编写
  报告: 方案论证 ←→ 硬件/软件: 可行性评估

Day2 并行:
  硬件: 传感器集成 ←→ 软件: 传感器驱动调试
  硬件: 电源优化 ←→ 软件: PID算法调试
  报告: 理论分析 ←→ 硬件/软件: 参数记录

Day3 并行:
  硬件: 可靠性测试 ←→ 软件: 发挥功能开发
  全员: 功能联调 ←→ 报告: 测试数据记录
```

### 8.2 依赖关系管理
```
硬件完成 → 软件调试 → 测试执行 → 报告数据
    ↓           ↓           ↓
  硬件报告   软件报告   测试报告
    ↓           ↓           ↓
          最终报告整合
```

### 8.3 沟通检查点
| 时间 | 检查内容 | 参与者 |
|------|---------|--------|
| 每天早上 | 前日进度回顾+今日计划 | 全员 |
| 每天中午 | 上午问题汇总+下午调整 | 全员 |
| 每天晚上 | 当日成果确认+风险评估 | 全员 |
| Day2晚上 | 基本功能确认(关键节点) | 全员 |
| Day3晚上 | 最终状态评估 | 全员 |

### 8.4 常见并行冲突解决
| 冲突场景 | 解决方案 |
|---------|---------|
| 硬件/软件抢同一引脚 | 提前约定引脚分配表 |
| 联调时互相干扰 | 分时调试,或使用独立串口 |
| 报告需要数据但还没测 | 报告先写框架,数据后补 |
| 发现硬件问题需要返工 | 评估影响范围,优先保证基本功能 |
| 某人任务提前完成 | 主动支援其他角色的紧急任务 |
